# SDD — 10.0 Commerce & Inventory Foundation

## 1. Objetivo técnico

Implementar a fundação modular de catálogo, estoque, pedidos e integrações omnichannel sem misturar domínio de inventário, pagamentos e conciliação. A plataforma permanece um monólito modular com contratos explícitos e pronta para extração futura de serviços.

## 2. Bounded contexts e dependências

```text
platform/
  identity, tenancy, RBAC, entitlements, audit, feature flags

catalog/
  products, variants, SKUs, cost policies

inventory/
  warehouses, inventory ledger, balance projection, reservations

orders/
  normalized orders, order items, lifecycle

channels/
  integrations, listings, mapping queue, inbound webhooks, outbound sync

financial-intelligence/
  order financial facts, COGS snapshot, analytical read models

payments/
  payment creation, provider state, refunds

reconciliation/
  settlement and cash reconciliation (phase 9A, deferred)
```

Dependências permitidas:

```text
Catalog → Platform
Inventory → Catalog + Platform
Orders → Catalog + Inventory + Platform
Channels → Orders + Catalog + Inventory + Platform
Financial Intelligence → Orders + Inventory + Platform
Payments ↔ Orders only through explicit contracts/events when needed
Reconciliation consumes Payments/Orders facts but does not own inventory
```

Dependências proibidas:

- Channel adapter lendo tabelas internas de Inventory diretamente.
- Payments alterando saldo físico diretamente.
- Financial Intelligence alterando pedido, estoque ou pagamento.
- UI decidindo autorização/entitlement final.

## 3. Dados e modelo relacional

### 3.1 Princípios

- `tenantId` em todas as entidades tenant-scoped.
- IDs UUID.
- Valores monetários `Decimal(18, 4)` internamente quando rateios puderem exigir precisão; exposição monetária arredondada conforme moeda.
- Quantidade `Decimal(18, 6)` para suportar unidade fracionada no futuro, mesmo se a UI inicial usar inteiros.
- Instantes em UTC.
- Soft archive para catálogo; ledger/eventos nunca sofrem delete físico.
- Names/enum de domínio centralizados, nunca strings espalhadas.

## 4. Concorrência e transações de inventário

### 4.1 Regra central

Nenhuma atualização de saldo é feita por read-modify-write em memória sem proteção.

Para reservar estoque:

1. iniciar transação PostgreSQL;
2. garantir ordenação estável das linhas por `skuId, warehouseId`;
3. bloquear `InventoryBalance` com `SELECT ... FOR UPDATE`;
4. validar `availableQuantity >= requestedQuantity`;
5. criar `InventoryReservation`;
6. criar `InventoryMovement` do tipo `RESERVATION`;
7. atualizar projeção de saldo;
8. criar OutboxEvent na mesma transação;
9. commit.

A primeira versão pode usar SQL explícito para o lock dentro de repository Prisma, preservando a fronteira de aplicação. Não usar lock distribuído Redis como substituto da integridade relacional do banco.

### 4.2 Movimentos e projeções

`InventoryMovement` é o ledger imutável. `InventoryBalance` é uma projeção transacional/materializada, atualizada na mesma transação para leituras rápidas. Uma rotina de reconciliação futura deve conseguir recalcular `InventoryBalance` a partir do ledger.

Todos devem usar Outbox transacional. Eventos de entrada de provider usam Inbox idempotente por `[provider, providerEventId]`.

## 6. Inbound channel intake

```text
POST /webhooks/channels/:provider
→ autenticação do provider
→ persistir ChannelWebhookInboxEvent
→ criar OutboxEvent
→ responder 200/202 rapidamente
→ worker busca detalhes com adapter
→ valida payload via Zod
→ normaliza pedido
→ aplica idempotência
→ reserva estoque
→ emite pedidos de sincronização
```

O limite de 200ms é objetivo operacional; a implementação deve medir e alertar p95/p99, não prometer um tempo impossível sob falhas de banco.

## 7. Outbound sync e proteção de APIs

Para cada `channel.inventory_sync.requested`:

1. formar chave coalescente `[tenantId, channelIntegrationId, listingId]`;
2. manter apenas o saldo mais recente pendente quando ainda não iniciado;
3. aplicar rate policy por provider e tenant;
4. usar retries exponenciais com jitter;
5. ao receber `429`, honrar `Retry-After` quando disponível;
6. usar circuit breaker por integração/provider;
7. registrar estado técnico sanitizado e permitir replay administrativo.

Não implementar 50 RPS como valor fixo global. Cada provider deve expor uma `RateLimitPolicy`, configurável e documentada conforme contrato oficial do canal.

## 8. Segurança e entitlements

- Tokens de canais ficam criptografados pelo serviço central de credenciais.
- Controllers tomam tenant do JWT.
- Entitlements são avaliados por guard/policy no backend.
- Platform admin vê saúde resumida, nunca tokens e payloads brutos.
- Webhooks validam assinatura/token e usam replay protection quando provider suportar.
- Logs não contêm PII desnecessária, tokens, URLs privadas, payload bruto ou documentos.

## 10. Observabilidade

- traceId API → Outbox → RabbitMQ → Worker.
- Métricas: intake latency, webhook duplicate rate, DLQ depth, sync latency, provider 429, circuit state, reservation conflict, negative stock prevention, export duration.
- Logs estruturados e sanitizados.
- Audit log para mudanças de SKU, custo, ajuste, mapping manual, reserva/liberação e feature entitlement.

## 11. Estratégia de testes

- Unitários: SKU policy, custo, transições de reserva, cálculo de available.
- Integração: transação com lock, idempotência de movimento, isolamento tenant.
- Contract: adapters de canal e schemas Zod.
- E2E: produto → entrada → reserva → cancelamento → fulfillment.
- Carga: múltiplas reservas concorrentes sobre o mesmo SKU.
- Segurança: tenant bypass, entitlement bypass, webhook inválido.
